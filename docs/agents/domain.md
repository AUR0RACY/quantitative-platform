# 领域文档

工程技能在探索代码库时应如何消费本仓库的领域文档。

## 探索前先读这些

- **`CONTEXT.md`** 在仓库根目录，或
- **`CONTEXT-MAP.md`** 在仓库根目录（如果存在）— 它指向每个上下文一个 `CONTEXT.md`。阅读与主题相关的每一个。
- **`docs/adr/`** — 阅读涉及你将要操作的领域的 ADR。

如果这些文件不存在，**静默继续**。不要提示缺失；不要建议提前创建。生产者 skill（`/grill-with-docs`）会在术语或决策实际确定时延迟创建它们。

## 文件结构

单上下文仓库（大多数仓库）：

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-xxx.md
│   └── 0002-xxx.md
└── src/
```

## 使用术语表的词汇

当你的输出中命名一个领域概念时（在 Issue 标题、重构提案、假设、测试名称中），使用 `CONTEXT.md` 中定义的术语。不要漂移到术语表明确避免的同义词。

如果你需要的概念不在术语表中，这是一个信号 — 要么你在发明项目不使用的语言（重新考虑），要么存在真正的空白（记录给 `/grill-with-docs`）。

## 标记 ADR 冲突

如果你的输出与现有 ADR 矛盾，显式指出而不是静默覆盖：

> _与 ADR-0007（xxx）矛盾 — 但值得重新讨论，因为……_
